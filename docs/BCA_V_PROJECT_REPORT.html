1. INTRODUCTION

The Hospital Management System (HMS) 2.0 is a web-based solution built with PHP 8 and SQLite, designed to streamline core hospital operations: patient registration, appointments, doctor management, billing, reporting, and system settings. It includes currency formatting with optional live conversion, printable and CSV exports with date/doctor/department filters, and CI-backed quality checks. The system emphasizes secure authentication, clean UI, and deployment readiness on Windows desktop (PHPDesktop) or as a web app.

The documentation adheres to the BCA V project format guidelines: Times New Roman (12 pt body, 14 pt headings, 16 pt titles), A4 page size with 1.5" left and 1" other margins, 1.5 line spacing, bottom-center page numbers, and figures/tables with captions.

2. OBJECTIVE

  • Digitize and centralize hospital data (patients, doctors, appointments, billing).
  • Provide fast, accurate billing with currency formatting and optional live conversion.
  • Enable filterable reporting (date, doctor, department) with printable and CSV exports.
  • Maintain data integrity via validation, foreign keys, and role-based access.
  • Ensure deployment simplicity using a portable SQLite database and scripts.

3. SYSTEM ANALYSIS

3.1 IDENTIFICATION OF NEED

Hospitals require integrated systems to reduce manual errors, accelerate patient throughput, and generate compliant reports. HMS addresses disparate records, delayed billing, and poor visibility by unifying operations under a single interface.

3.2 PRELIMINARY INVESTIGATION

Stakeholders: Admin, Doctors, Receptionists, Accountants. Pain points: duplicate entries, delayed appointment confirmations, inconsistent currency handling, manual reporting. Proposed solution: a modular HMS with validations, consistent currency formatting, and automated reports.

3.3 FEASIBILITY STUDY

3.3.1 TECHNICAL FEASIBILITY

Built on PHP 8 and SQLite—widely supported and easy to deploy. Frontend uses Bootstrap 5 and standard JS. No external servers required for database; works offline in desktop packaging.

3.3.2 OPERATIONAL FEASIBILITY

Clear UI, role-based access, and validation reduce training time. Printable/CSV reports meet audit needs. Currency defaults and live conversion align with international contexts.

3.3.3 ECONOMIC FEASIBILITY

Open-source stack lowers costs. SQLite avoids database licensing. Minimal hardware required; scales from single desktop to LAN deployment.

3.4 PROJECT PLANNING

Phased delivery: schema + server setup → auth + core CRUD → billing + currency → reporting (CSV/PDF) → deployment/CI.

3.6 SOFTWARE REQUIREMENT SPECIFICATION (SRS)

3.6.1 FUNCTIONAL REQUIREMENTS

  • Manage patients, doctors, appointments, billing, payments.
  • Currency formatting with default + optional live conversion.
  • Reports: CSV and printable PDF with date/doctor/department filters.
  • Authentication with roles (admin, doctor, receptionist).
  • System settings persisted in system_settings (e.g., currency_default, live_conversion).
  • Auto-billing hook on completed appointment (consultation fee tracking).

3.6.2 NON-FUNCTIONAL REQUIREMENTS

  • Usability: Bootstrap UI, responsive tables.
  • Reliability: SQLite ACID, prepared statements, CI checks.
  • Performance: Lightweight stack, minimal dependencies.
  • Security: Session handling, no WWW-Authenticate popups, role checks.
  • Portability: Single-file SQLite DB for easy backup and migration.
  • Maintainability: Modular PHP classes and API endpoints per feature.

3.7 SYSTEM SPECIFICATION

  • Software: PHP 8.2+, SQLite3, Bootstrap 5, Font Awesome.
  • Hardware: Standard Windows 10/11 PC; 2–4 GB RAM; 1 GB disk.
  • Browser: Chrome/Edge latest recommended for printing.
  • CI: GitHub Actions (lint, DB init, server boot, HTTP checks).

4. SYSTEM DESIGN

4.1 MODULARIZATION DETAILS

Modules: Patients, Doctors, Appointments, Billing, Reports, Settings. Each module has corresponding API endpoints in /api, class logic in /classes, and UI/js in /assets/js.

4.2 DATA INTEGRITY AND CONSTRAINTS

  • Foreign keys: appointments.patient_id → patients.id; appointments.doctor_id → doctors.id; billing.appointment_id → appointments.id.
  • Constraints: NOT NULL, UNIQUE (e.g., users.username, billing.bill_number), CHECKs for enums.
  • Validation: server-side (PHP) + client-side (JS) where applicable.

6. SYSTEM SECURITY MEASURES

  • Session-based auth with role checks; JSON errors (no browser auth prompts).
  • Prepared statements for SQL; headers prevent caching of sensitive data.
  • Settings persisted in system_settings; currency defaults controlled centrally.
  • CI-only auth bypass via env var; never enabled in production.
  • Password hashing (bcrypt); role-based checks for sensitive operations.
  • Log files for diagnostics; avoid exposing stack traces to end users.
